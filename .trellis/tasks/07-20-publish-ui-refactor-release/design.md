# 发布全局 UI 重构版本：技术设计

## 边界

本任务只更新公开源码、项目说明和现有 `v2.0` Release。应用运行代码、数据库和用户学习数据不在变更范围内。构建产物继续留在 Git 忽略的 `dist/`，仅作为 GitHub Release 资产上传。

## 发布一致性

发布完成后的唯一版本基线为最终 `main` 提交：

- 本地 `main`、`origin/main` 和 annotated tag `v2.0` 的解引用提交一致。
- GitHub Release `v2.0` 绑定该标签，标题保持“我爱学习 v2.0”。
- Release 资产为本次已验证的 `I-Love-Learning-Portable.zip`。
- README、GitHub Description 与 Release 正文使用一致的产品定位，但分别承担项目概览、仓库摘要和版本变更说明。

## 内容设计

- README 首屏突出“本地优先、多项目学习、GUI/Web 双模式、响应式蓝靛界面”，详细安装、数据安全和开发说明继续留在后续章节。
- GitHub Description 使用一句话：`本地优先的多项目题库、间隔复习、模拟考试与学习进度平台`。
- Release 正文说明本次 UI 重构、导航与首页、答题体验、知识树、移动端、Portable/Docker 能力及质量验证，并附 ZIP 的 SHA-256。

## 数据流

1. 在本地完成 README 与 Trellis 发布记录，运行发布门禁并提交。
2. 推送 `main`，确认远端分支指向最终提交。
3. 重新创建 annotated tag `v2.0` 并强制更新远端同名标签。
4. 更新仓库 Description 和现有 Release 元数据。
5. 替换旧 ZIP 资产，重新下载远端资产并计算 SHA-256。
6. 核对 GitHub API 返回的标签、正文、资产和摘要，确认工作区干净。

## 凭据与安全

- Git 操作复用已配置的 Git Credential Manager，不输出凭据。
- GitHub 元数据与资产优先通过受认证 API 更新；若无法安全取得 API 会话，则使用现有已登录浏览器完成相同操作。
- 上传范围严格限制为 `dist/I-Love-Learning-Portable.zip`，不上传 `.sha256`、数据库、题库、导入原件或日志。
- 发布前运行公开仓库检查；发布后下载远端 ZIP 到临时目录校验，不覆盖本地构建。

## 回滚

- 操作前记录旧标签解引用提交、Release ID、资产 ID、旧正文与旧资产摘要。
- 在新资产上传成功并可下载前，不删除本地新 ZIP。
- 若标签更新失败，恢复旧标签目标；若元数据更新失败，使用保存的旧正文恢复。
- 若资产替换失败，立即重试上传，确保 Release 不长期处于无资产状态；必要时恢复预先保存的旧资产。

## 取舍

复用 `v2.0` 会重写同名标签和资产，既有下载链接的内容会变化；这是用户明确选择的发布策略。优点是维持当前版本号与既有下载入口，代价是历史可追溯性弱于创建 `v2.1`，因此必须用提交对齐和摘要校验保证当前状态明确。
