# 技术设计

## Architecture

- 保持 Flask/Jinja 服务端渲染和原生 JavaScript；改动集中在共享页面壳层、模板语义类、静态样式和少量导航交互。
- CSS 按 Token、基础、组件、布局/页面、响应式的依赖顺序组织。旧选择器在完成迁移后删除，避免新旧系统长期叠加。
- 共享 Jinja 宏仅用于消除重复的页面头、状态标签、分页或空状态结构；不改变表单字段、路由参数或服务端处理。

## Design Contracts

- 颜色只能通过语义变量消费：canvas/surface/text/border/primary/success/warning/danger/info，并为浅深主题分别映射。
- 页面结构统一为页面头、主行动、内容分区和辅助摘要。主要 CTA 使用实心深蓝靛；其他操作依次使用 tonal、outline、text 或 icon。
- 桌面导航分为学习、内容和系统；手机底栏固定四个核心入口与一个“更多”抽屉。直接 URL 与模块权限判断不变。
- 手机复杂表格通过 CSS/语义标记转成记录列表；桌面仍使用真正的表格语义和紧凑密度。
- 图标继续使用内联 SVG，统一视口、尺寸和描边；不引入第三方运行时资产。

## Homepage Data Flow

- 复用现有首页上下文，不新增持久化字段。
- 主行动按活动会话、到期复习、当前周计划、自由练习顺序选择；若某模块未启用则跳过对应候选。
- 趋势、准备度、计划和项目统计保留现有数据来源，只改变展示优先级与目标链接。

## Compatibility And Rollback

- 不执行数据库迁移，已有项目、题目、会话、设置和学习数据不受影响。
- 保持现有元素 ID、表单 name、接口 URL 和关键 JavaScript hook；新增类名只负责呈现。
- 分三阶段提交可验证改动；若某阶段出现功能回归，可回退该阶段模板/CSS，而无需恢复数据。
- Windows GUI/Web 和 Docker 共用静态资源，因此所有平台差异只通过既有运行方式处理。

## Trade-offs

- UI 数据库建议中的在线字体、夸张大标题和 GSAP 动效与离线、沉静、无依赖约束冲突，因此采用系统字体和克制原生过渡。
- 不通过继续追加覆盖快速完成，而是重建设计系统；初期修改面较大，但能降低颜色漂移和 `!important` 债务。
